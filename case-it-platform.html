    Описание проекта

    Контекст и аудитория

    Цель проекта

    Построить клиентскую платформу для управления задачами и статусами: единые правила работы, прозрачные переходы и быстрые согласования.

    Целевая аудитория

    Супервайзеры, операционные менеджеры и сотрудники поддержки, которым нужно видеть статус задач и быстро передавать работу между ролями.

    Конкуренты

    Конкурент Сильные стороны Слабые стороны Вывод
    Jira Service Гибкая система задач Сложные настройки и перегруз UI Нужен упрощённый сценарий для клиента
    Asana Чистый интерфейс Слабая детализация статусов Усилить контроль и маршрутизацию
    Monday Гибкость колонок Слишком визуально шумный Сделать строгую иерархию

    Постановка задачи

    Цели, боли, гипотезы, метрики

    Цели

    • Сократить время согласования задач минимум на 25%.
    • Повысить прозрачность статусов для всех ролей.
    • Снизить количество ошибок при передаче задач.

    Боли

    • Нет единого стандарта статусов.
    • Потеря задач между ролями.
    • Сложный поиск нужной информации.

    Гипотезы

    • Единый статус‑флоу снизит ошибки передачи на 20–30%.
    • Карточный интерфейс ускорит поиск задач.
    • Ролевые фильтры снизят время согласования.

    Метрики успеха

    • Time‑to‑Approve.
    • Количество возвратов задач.
    • Скорость поиска задачи.

    Команда и роли

    Scrum‑команда

    Состав

    • Product Manager, аналитик.
    • UX/UI дизайнер (я).
    • 3 Frontend, 2 Backend, QA.

    Моя роль

    • Карта процессов и user flows.
    • Информационная архитектура.
    • UI‑гайдлайн и контроль внедрения.

    Функционал

    Ключевые фичи

    • Карточный список задач по ролям.
    • Статусы с чёткими переходами.
    • Шаблоны сценариев “создать → согласовать → выполнить”.
    • История действий и комментарии.

    Сценарий

    Менеджер создаёт задачу, назначает ответственного и сразу видит статус. Ответственный получает уведомление, закрывает задачу и оставляет комментарий.

    Ограничения

    Временные и технические рамки

    • Срок: 9 недель.
    • Нельзя менять базовую систему прав доступа.
    • Поддержка мобильной версии и светлой темы.

    Процесс

    Исследования, артефакты, тесты

    Research

    • 12 интервью с менеджерами и операторами.
    • Аудит старых сценариев и логов.
    • Workshops по статусам и переходам.

    UX‑артефакты

    • Service blueprint по цепочке задач.
    • User flows по ролям.
    • Wireframes ключевых сценариев.

    Тестирование

    • 2 раунда тестов с пользователями.
    • Сокращение шагов на 30% по итогам итераций.

    Визуализация

    UI‑решения

    • Светлая тема с высокой читаемостью.
    • Статусы и индикаторы как ключевые сигналы.
    • Единые компоненты для задач и карточек.

    В разработке

    Front‑review

    • Согласование переходов статусов.
    • Проверка соответствия UI‑гайдам.
    • Тесты интеграции с правами доступа.

    Результаты

    Пользовательский эффект

    • Time‑to‑Approve: −28%.
    • Ошибки передачи задач: −24%.
    • Снижение ручных уточнений: −19%.
    • Рост прозрачности статусов: +30%.

    Фидбэк

    Выводы

    • Ключевым фактором стало единое ядро статусов.
    • Ролевые фильтры заметно ускорили работу.
    • Следующий шаг — автоматизация SLA.
