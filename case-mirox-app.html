Решения и процесс

    Описание проекта

    Контекст и аудитория

    Цель проекта

    Спроектировать мобильное приложение для сотрудников «в поле», где выполнение задач должно занимать минимум времени, работать в условиях слабой связи и оставаться понятным с первого экрана.

    Целевая аудитория

    Технические специалисты и супервайзеры, которые закрывают задачи на объектах и фиксируют результат через фото, чек‑листы и короткие комментарии.

    Конкуренты

    Конкурент Сильные стороны Слабые стороны Вывод
    PlanRadar Mobile Многофункциональность Перегруженный UX для быстрого ввода Упростить сценарии до 2–3 шагов
    Fieldwire Хорошие офлайн‑сценарии Сложная навигация по задачам Сделать прямой доступ к активным задачам
    Asana Mobile Чистый интерфейс Недостаток отраслевых сценариев Усилить блок проверки и статусов

    Постановка задачи

    Цели, боли, гипотезы, метрики

    Цели

    • Сократить время выполнения одной задачи минимум на 25%.
    • Снизить долю незавершённых задач.
    • Обеспечить устойчивую работу офлайн.

    Боли пользователей

    • Слишком много шагов для фиксации результата.
    • Сложно найти нужную задачу среди списка.
    • Проблемы при слабом интернете.

    Гипотезы

    • Если вынести активные задачи на главный экран — время поиска сократится на 40%.
    • Если сделать “быстрые действия” — количество завершённых задач увеличится.
    • Если добавить офлайн‑режим — снизится число ошибок и потерь данных.

    Метрики успеха

    • Среднее время закрытия задачи.
    • Процент задач, завершённых без возврата.
    • Частота использования офлайн‑режима.

    Команда и роли

    Scrum‑команда

    Состав

    • Product Manager, Scrum Master.
    • UX/UI дизайнер (я).
    • 2 Mobile dev (iOS/Android), Backend, QA.

    Моя роль

    • Полевые интервью и формирование сценариев.
    • Проектирование мобильных потоков.
    • UI‑гайдлайн и контроль адаптивности.

    Функционал

    Ключевые фичи

    • Список активных задач с приоритетами.
    • Быстрое подтверждение через чек‑лист.
    • Фото‑фиксация и короткие комментарии.
    • Офлайн‑режим с синхронизацией.
    • Статусы и история действий.

    Сценарий

    Пользователь открывает приложение, выбирает задачу, делает фото результата, отмечает чек‑лист и закрывает задачу в 2–3 шага. Если сети нет — данные сохраняются и отправляются при восстановлении связи.

    Ограничения

    Временные и технические рамки

    • Срок: 8 недель, MVP для пилота.
    • Поддержка старых устройств и слабой сети.
    • Ограничения на размер и хранение фото.

    Процесс

    Исследования, артефакты, тесты

    Сфокусировался на mobile‑first процессе: интервью в «полевых» условиях и быстрые прототипы для проверки удобства одной рукой.

    Research

    • 8 интервью с полевыми специалистами.
    • Наблюдения на объектах (shadowing).
    • Анализ типовых ошибок ввода.

    UX‑артефакты

    • User flows для “принять → выполнить → закрыть”.
    • Карты контекста мобильной работы.
    • Прототипы быстрых действий.

    Тестирование

    • 3 итерации прототипов, 10 тестов.
    • Оптимизация экранов под удержание одной рукой.

    Визуализация

    UI‑решения

    • Монохромный интерфейс для снижения визуального шума.
    • Крупные CTA и акцент на статусах.
    • Минимальное количество экранов до результата.

    В разработке

    Front‑review

    • Согласование мобильных переходов с dev‑командой.
    • Тесты офлайн‑режима и синхронизации.
    • Проверка стабильности на разных девайсах.

    Результаты

    Пользовательский и бизнес‑эффект

    • Среднее время закрытия задачи: −29%.
    • Рост завершённых задач: +22%.
    • Ошибки ввода данных: −31%.
    • Частота использования офлайн‑режима: 46% сессий.

    Фидбэк

    Выводы и сложности

    • Главное — убрать лишние поля и дать быстрый путь.
    • Быстрые действия оказались ключевым фактором принятия.
    • Следующий шаг — голосовой ввод и авто‑распознавание.
